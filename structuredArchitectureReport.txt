## 7. Proposed Backend Architecture (Industrial IoT Focus)

**Ingestion & Edge**
- Edge gateways push sensor readings via MQTT/HTTP.
- Use a gateway ingestion service (MQTT broker + HTTP fallback).
- Validate data on ingress for sanity checks (range validation, schema checks).

**Stream Processing**
- Stream processor for:
  - Aggregations (min/max/avg AQI)
  - Alert rule evaluation
  - Anomaly detection
- Publish alert events into an alerts pipeline.

**Storage**
- Time‑series DB for sensor readings (e.g., TimescaleDB, Influx, ClickHouse).
- Relational DB for metadata (devices, sensors, rules, alerts, api keys).
- Object storage for reports (PDFs).

**Alert & Rules Engine**
- Rule evaluator consumes streams + thresholds.
- Supports actions: notification, warning, alert log.
- Store alerts and transitions (active → resolved/dismissed).

**Device Health Monitoring**
- Heartbeat tracking service.
- Status derived from last heartbeat + signal/battery thresholds.
- Expose aggregated health stats to frontend.

**Bulk Sync**
- Separate batch ingestion endpoint for offline data uploads.
- Packet validation pipeline.
- Batch logs stored with metrics (received/rejected/duration).

**Public API Gateway**
- API key authentication.
- Rate limit enforcement.
- Separate public endpoints (AQI, sensors, historical data).

**Reporting**
- Report generation workers (async jobs).
- Queries from time‑series DB + alert summaries.
- Store results and provide download links.

**Observability & Ops**
- Service health endpoint for system status page.
- Metrics exporter for CPU/memory/uptime.
- Audit logs for rule changes and API key events.
